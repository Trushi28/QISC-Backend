→ Longer compilation time
→ IR is heavily transformed
Compilation Budget Allocation:

Compiler allocates:
  - Parsing: 1s (fixed)
  - Profile loading: 0.5s (fixed)
  - Hot path optimization: 6s (most bang for buck)
  - Warm path optimization: 1.5s
  - Cold path optimization: 0.5s
  - Code generation: 0.5s (fixed)
11.12 Pragmas for Multi-Stage Compilation
Progressive compilation hints:
#pragma stage:1
// First compilation: just get it working
#pragma optimize:none
#pragma profile:collect

#pragma stage:2  
// Second compilation: start optimizing
#pragma optimize:balanced
#pragma profile:refine

#pragma stage:3+
// Third+ compilation: go aggressive
#pragma optimize:aggressive
#pragma profile:validate
Automatic stage progression:
qisc build --auto-stage

Automatically:
  1. First compile with stage:1 pragmas
  2. Run and collect profile
  3. Recompile with stage:2 pragmas
  4. Verify improvement
  5. Recompile with stage:3 pragmas
  6. Continue until convergence

Integration: How Pragmas Fit the Overall Theory
The Three-Way Handshake
        Developer Intent              Observed Reality           Compiler Knowledge
           (Pragmas)     ←→  Negotiation  ←→    (Profiles)    ←→   (Algorithms)
                                    ↓
                          Optimized Living IR
                                    ↓
                          Converged Binary
Example of Three-Way Interaction:
Developer: #pragma optimize:latency
           "I care about latency"

Profile:   Function called rarely, but takes 100ms when called
           "Actually, this IS latency-sensitive despite low frequency"

Compiler:  "Okay, I have algorithms for latency optimization:
           - Reduce allocation (memory latency)
           - Inline critical path (call overhead)
           - Prefetch data (memory latency)
           All three parties agree: optimize for latency"

Result:    Heavily optimized, low-latency implementation
Counter-Example of Conflict:
Developer: #pragma optimize:memory
           "I think memory is constrained"

Profile:   Memory usage is 10MB, system has 16GB
           Speed is 50% slower than needed
           "Memory is NOT the bottleneck"

Compiler:  "Profile shows speed is critical, not memory.
           Warning: Ignoring optimize:memory
           Applying optimize:speed instead
           Suggestion: Remove this pragma or change to optimize:speed"

Result:    Speed-optimized, developer learns about actual constraints

Why This Pragma Theory Matters

Developer Guidance Without Lock-In: Pragmas guide but don't force
Reality Wins: Profile can override incorrect pragmas
Continuous Learning: Pragma + Profile creates feedback loop
Gradual Refinement: Start with rough pragmas, refine over time
Conversation, Not Commands: Compiler talks back to developer
Type Safety for Optimization: Pragmas make optimization intent explicit and checkable
